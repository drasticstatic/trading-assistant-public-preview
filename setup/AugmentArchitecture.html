Three surfaces

Same ecosystem, three different front doors

The simplest way to orient a new teammate is to separate the main surfaces first, then talk about the shared machinery underneath.

VS Code Agent

Editor-native builder bench for tight iteration, file edits, and review loops without leaving the IDE.

Best at: in-editor build work

Auggie CLI

Terminal-native builder bench for shell-heavy work, repo inspection, and command-driven implementation.

Best at: terminal-first workflows

Intent

Spec-driven orchestration layer for planning, delegation, specialist lanes, and cleaner handoffs.

Best at: coordination + task routing

Workflow pattern

Why Intent feels team-shaped

The most useful pattern is not “pick one super-agent.” It is using narrower roles together so each one has less ambiguity about what good looks like.

Coordinator

Writes the plan, breaks work into scoped tasks, and keeps the spec or note structure clean.

Implementor

Builds one lane cleanly, runs scoped verification, and reports exact files changed.

Reviewer / Verifier / UI

Checks correctness, presentation, and regression risk without collapsing everything into one thread.

What seems clear vs open

Two buckets that keep the guide honest

Clear from docs + usage

  • Augment exposes multiple user-facing surfaces on purpose.
  • Rules, tools, and codebase context meaningfully shape behavior.
  • Specialists/subagents are a real supported pattern, not just a metaphor.
  • Shared repo understanding is not the same thing as shared chat history.

Still worth watching

  • The exact internal model-routing contract is not fully exposed in the public docs reviewed for the markdown guide.
  • Your chosen model likely matters a lot, but it may not be the only system involved in every behind-the-scenes step.
  • That is why the safest phrasing is “lead reasoning path,” not “only model ever used.”
